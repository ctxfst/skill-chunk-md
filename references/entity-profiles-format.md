# Entity Profiles Format Reference

Complete reference for the **entity profiles** JSON schema used in the CtxFST pipeline. Entity profiles are a derived, intermediate data structure generated from a core CtxFST export. They aggregate chunk-level information to create rich, embeddable representations for each canonical entity.

**This format is not part of the core CtxFST specification** ([`ctxfst-spec.md`](../../references/ctxfst-spec.md)). It is a derived pipeline artifact. The core schema remains unchanged; only the reference scripts produce and consume this structure.

---

## 1. What are Entity Profiles?

While the core CtxFST schema defines entities merely as a canonical inventory (ID, name, type, aliases), the **entity profiles pipeline** reverse-links chunks to entities to build a comprehensive view of how each entity is actually used in the text.

This derived format is typically generated by `scripts/build_entity_profiles.py` and serves as the input for downstream processes, such as generating an entity similarity graph or populating a vector database with rich entity embeddings.

---

## 2. Root Structure

The JSON document contains two top-level keys: `meta` and `entities`.

```json
{
  "meta": { ... },
  "entities": [ ... ]
}
```

---

## 3. Metadata (`meta`)

The `meta` object contains information about the build process and parameters used to generate the profiles.

| Field | Type | Description |
|-------|------|-------------|
| `input` | string | Path to the source CtxFST export JSON |
| `source_type` | string | Always `"ctxfst-export"` |
| `profile_type` | string | Always `"entity-profiles"` |
| `entity_count` | integer | Number of entities in the `entities` array |
| `max_contexts` | integer | The maximum number of chunk contexts retained per entity |
| `top_keywords` | integer | The maximum number of derived keywords retained per entity |
| `dangling_chunk_entity_refs` | string[] | Array of entity IDs referenced by chunks but missing from the canonical entities list |

---

## 4. Entity Object (`entities[]`)

Each item in the `entities` array is a comprehensive profile of a single concept, combining its canonical definition with aggregated usage data.

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | The canonical `entity:{name}` identifier. |
| `name` | string | Human-readable canonical name. |
| `type` | string | Concept classification (e.g., `skill`, `framework`, `tool`). |
| `aliases` | string[] | Alternative spellings or abbreviations. |
| `mentioned_chunks` | string[] | Array of chunk IDs where this entity was mentioned. |
| `mention_count` | integer | Total number of chunks mentioning this entity (length of `mentioned_chunks`). |
| `tags` | string[] | Aggregated, deduplicated `tags` from all linked chunks. |
| `contexts` | string[] | Aggregated `context` summaries from linked chunks (capped by `max_contexts`). |
| `content_excerpts` | string[] | Aggregated, truncated `content` blocks from linked chunks. |
| `related_entities` | string[] | Names of other entities that co-occur with this entity in the same chunks. |
| `keywords` | string[] | Frequent lightweight term extracts from linked chunk text (capped by `top_keywords`). |
| `representation` | string | A concatenated, multiline text block designed specifically for embedding models. |

---

## 5. The `representation` Field

The `representation` string is the most important output for downstream semantic/embedding pipelines. It fuses the entity's canonical identity with its contextual usage across the entire corpus.

This text is specifically formatted to be passed to LLM embedding models (like OpenAI `text-embedding-3-*` or HuggingFace models) to compute dense vectors.

**Example `representation`:**
```text
name: FastAPI
type: framework
tags Python Backend API
context Python programming skills focusing on data pipelines and REST APIs using FastAPI and Pandas
content ### Python

I use Python daily for building data pipelines and REST APIs. My preferred frameworks are FastAPI for APIs and Pandas for data processing.

Key areas of expertise:
- Async programming with asyncio
- Performance profiling and optimization
- Testing with pytest
related Pandas Python
```

### Representation prefix tokens (for parsers)

If you are building a parser or reimplementing the format in another language, the `representation` text uses the following line prefixes. Each line is either a key-value line or a continuation of the previous block.

| Prefix | Meaning | Example |
|--------|---------|---------|
| `name:` | Canonical entity name | `name: FastAPI` |
| `type:` | Entity type | `type: framework` |
| `aliases ` | Space-separated aliases (no colon) | `aliases python3 py` |
| `tags ` | Space-separated tags from chunks | `tags Python Backend API` |
| `context ` | One chunk context string (may span multiple lines until next prefix) | `context Python programming skills...` |
| `content ` | Truncated chunk content (may span multiple lines until next prefix) | `content ### Python\n\nI use Python...` |
| `related ` | Space-separated names of co-mentioned entities | `related Pandas Python` |

Order of lines is not guaranteed across versions; prefer parsing by prefix rather than by position.

---

## 6. Pipeline Architecture

1. **Source**: Write markdown (`.ctxfst.md`).
2. **Export**: Run `export_to_lancedb.py` → produces `chunks.json`.
3. **Profile**: Run `build_entity_profiles.py chunks.json` → produces **`entity-profiles.json`**.
4. **Graph**: Run `build_entity_graph.py entity-profiles.json` → produces `entity-graph.json` (computes similarity edges).
